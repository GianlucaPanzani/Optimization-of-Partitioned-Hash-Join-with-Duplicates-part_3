# Optimization of Partitioned Hash Join with Duplicates - Module 2

This project contains a sequential baseline and four parallel variants of the full partitioned hash join with duplicates required by Module 2.

## Project layout

- `src/`: source files, scripts, Makefile, and generated executables
- `src/grid/`: benchmark grid configurations
- `src/runners/`: Slurm runner used by the benchmark launcher
- `src/results/`: CSV files generated by the executables

## Build

Move into `src/` before compiling:

```bash
cd src
```

Compile one executable at a time with `make`:

```bash
make hashjoin_seq
make hashjoin_par_p
make hashjoin_par_pj
make hashjoin_par_pj_wb
make hashjoin_par_pj_wb_map
```

## Common executable interface

All executables use the same command-line format:

```bash
./<executable> -nr <NR> -ns <NS> -seed <SEED> -max-key <MAX_KEY> -p <P> --partition-threads <PT> --join-threads <JT>
```

Parameters:

- `-nr`: number of records in relation `R`
- `-ns`: number of records in relation `S`
- `-seed`: deterministic seed used for data generation
- `-max-key`: keys are generated in `[0, max-key)`
- `-p`: number of partitions, expected to be a power of two
- `--partition-threads`: number of threads used for the partition phase
- `--join-threads`: number of threads used for the join phase

Each run prints:

- `join_count`
- `checksum1`
- `checksum2`

and appends one row to `results/<executable>.csv`.

## How to execute each executable

### `hashjoin_seq`

Sequential reference implementation.

```bash
./hashjoin_seq -nr 50000000 -ns 50000000 -seed 13 -max-key 1000000 -p 256 --partition-threads 1 --join-threads 1
```

### `hashjoin_par_p`

Parallel partitioning, sequential join.

```bash
./hashjoin_par_p -nr 50000000 -ns 50000000 -seed 13 -max-key 1000000 -p 256 --partition-threads 8 --join-threads 1
```

### `hashjoin_par_pj`

Parallel partitioning and parallel join.

```bash
./hashjoin_par_pj -nr 50000000 -ns 50000000 -seed 13 -max-key 1000000 -p 256 --partition-threads 8 --join-threads 8
```

### `hashjoin_par_pj_wb`

Parallel partitioning and parallel join with workload-balanced partition assignment.

```bash
./hashjoin_par_pj_wb -nr 50000000 -ns 50000000 -seed 13 -max-key 1000000 -p 256 --partition-threads 8 --join-threads 8
```

### `hashjoin_par_pj_wb_map`

Most optimized variant in this repository: workload-balanced join plus flatter partition counters/offsets and a custom flat count table in the join build phase.

```bash
./hashjoin_par_pj_wb_map -nr 50000000 -ns 50000000 -seed 13 -max-key 1000000 -p 256 --partition-threads 8 --join-threads 8
```

## Benchmark automation

The benchmark launcher is executed from `src/` with:

```bash
./runners/benchmark.sh <executable> grid/<grid_filename> <n_runs>
```

Examples:

```bash
./runners/benchmark.sh hashjoin_seq grid/seq_grid_search.sh 3
./runners/benchmark.sh hashjoin_par_p grid/par_grid_search.sh 3
./runners/benchmark.sh hashjoin_par_pj grid/par_grid_search.sh 3
./runners/benchmark.sh hashjoin_par_pj_wb grid/par_grid_search.sh 3
./runners/benchmark.sh hashjoin_par_pj_wb_map grid/par_grid_search.sh 3
./runners/benchmark.sh weak_scaling grid/weak_scaling_grid_search.sh 3
```

What the script does:

- cleans the selected target and its previous CSV
- recompiles the selected executable with `make -B`
- loads the grid file from `src/grid/`
- submits one Slurm job per configuration through `runners/run_with_slurm.sh`
- stores logs in `src/out/` and `src/err/`
- appends results to `src/results/<executable>.csv`

### Available grid files

- `grid/seq_grid_search.sh`: grid for the sequential baseline
- `grid/par_grid_search.sh`: grid for the parallel implementations
- `grid/weak_scaling_grid_search.sh`: grid dedicated to the weak scalability analysis

Each grid file defines:

- fixed `N_VALUE` and `P_VALUE`
- arrays of `SEED_VALUES`
- arrays of `MAX_KEY_VALUES`
- arrays of `PARTITION_THREAD_VALUES`
- arrays of `JOIN_THREAD_VALUES`

## Correctness notes

- For very small inputs (`NR <= 500` and `NS <= 500`), the programs also run a naive verifier and print the naive results.
- For larger inputs, correctness is checked by comparing `join_count`, `checksum1`, and `checksum2` across the sequential and parallel executables.
